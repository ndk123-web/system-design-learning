# Valet Key Pattern (Cloud Data Management)

## One-line Summary

Valet Key Pattern allows a client to directly upload or download data from cloud storage using a temporary, restricted URL generated by the application server, without routing large data through the server.

## 1. WHAT (What is Valet Key?)

Valet Key is a **temporary, limited-access permission** given to a client so that it can directly access a specific resource (like a file in cloud storage) for a short period of time.

- The server **does not transfer the data**
- The server **only decides permission**
- The storage service **enforces security**

Think of it as:

- NOT a password
- NOT permanent credentials
- A **signed, time-bound URL**

---

## 2. WHY (Why Valet Key exists?)

### Problem without Valet Key

Traditional flow:

```

Client → App Server → Cloud Storage

```

Issues at scale:

- App server becomes a **bandwidth bottleneck**
- High CPU and memory usage
- Slower response time
- Higher infrastructure cost
- Difficult horizontal scaling
- Larger attack surface

Reality:

> Application servers are bad at moving large files.
> Cloud storage is optimized for it.

---

## 3. HOW (How Valet Key works)

### High-level idea

```

Client → Server (asks permission)
Server → Client (gives temporary access)
Client → Storage (direct upload/download)

```

The **server never sees the file data**.

---

## 4. COMPLETE FLOW (Step-by-step)

### Step 0: Server Setup (Important)

The server already has **real cloud credentials**:

- AWS IAM
- GCP Service Account
- Azure Storage Key

⚠️ These credentials are NEVER sent to the client.

---

### Step 1: Client → Server (Request Permission)

Frontend sends metadata only:

```json
POST /request-upload
{
  "filename": "profile.jpg",
  "contentType": "image/jpeg"
}
```

Server checks:

- Is user authenticated?
- Is file type allowed?
- Is size limit acceptable?

No file upload yet.

---

### Step 2: Server → Storage (Create Valet Key)

Server asks storage:

> “Give me a URL that allows uploading `profile.jpg`
> valid for 5 minutes
> only for PUT operation”

Storage generates a **cryptographically signed URL**.

---

### Step 3: Server → Client (Return Valet Key)

Server responds:

```json
{
  "uploadUrl": "https://storage.cloud/bucket/profile.jpg?signature=XYZ&expires=5min"
}
```

This URL **is the Valet Key**.

---

### Step 4: Client → Storage (Direct Upload)

Frontend uploads file directly:

```http
PUT signed-upload-url
Body: image bytes
```

🔥 Server is completely bypassed.

---

### Step 5: Storage Verification (Automatic)

Storage validates:

- Signature authenticity
- Expiry time
- Allowed operation (PUT / GET)
- Allowed object path

If anything fails:

```
403 Forbidden
```

Server is not involved.

---

## 5. FLOW DIAGRAM

```
Client ──request upload──▶ Server
Client ◀─signed URL────── Server
Client ──upload file────▶ Cloud Storage
```

---

## 6. LOGIC (Why this pattern works)

### Separation of responsibility

| Component | Responsibility                 |
| --------- | ------------------------------ |
| Server    | Auth, validation, permission   |
| Storage   | Secure, scalable data transfer |
| Client    | Upload/download                |

Result:

- Server stays lightweight
- Storage handles heavy I/O
- System scales naturally

---

## 7. SECURITY MODEL (Very Important)

Valet Key is secure because it is:

- Time-limited (minutes)
- Resource-limited (single file)
- Operation-limited (PUT or GET)
- Scope-limited (specific path)

If URL leaks:

- Very small blast radius
- Automatically expires

---

## 8. WHERE THIS IS USED (Real Systems)

- Profile image uploads
- Video uploads (YouTube-like systems)
- Document storage (Drive, Dropbox)
- Log ingestion
- ML dataset uploads
- IoT data pipelines

---

## 9. COMMON MISTAKES

❌ Long expiry time (hours)
❌ Full bucket access
❌ Server still proxying data
❌ No file type or size limits

These lead to security and cost issues.

---

## 10. CONCEPT RULES (Remember Forever)

- Servers should make decisions, not move data
- Temporary access is safer than permanent credentials
- Offload heavy data paths early
- Cloud storage is part of your architecture, not just storage

---

## FINAL TAKEAWAY

Valet Key Pattern improves:

- Scalability
- Performance
- Security
- Cost efficiency

by letting clients **directly access cloud storage using temporary, restricted permissions generated by the server**.
